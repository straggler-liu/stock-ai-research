# 全品种投研决策平台（优化版）开发手册 v3.0

> 目标：在你提供的 v2.0 架构基础上，将“研究能力”与“交易执行能力”解耦，形成可验证、可追溯、可灰度上线的工程化系统。  
> 适用范围：A股、港股、美股、境内ETF、QDII ETF、公募基金、REITs。

---

## 0. 先回答核心问题：能否“保证高胜率”？

**结论：不能保证。**
任何量化/半量化投研系统都无法保证高胜率，原因包括：
1. 市场非平稳（Regime shift），历史有效规则可能失效；
2. 数据时滞、口径差异、异常点会放大误判；
3. 流动性与冲击成本会侵蚀纸面收益；
4. 黑天鹅与政策风险不可消除；
5. “高胜率”不等于“高收益”，需看盈亏比、回撤、容量和稳定性。

**可承诺的只有：**
- 决策流程标准化；
- 风险暴露可度量；
- 策略有效性可回测与可审计；
- 通过持续迭代提高长期期望收益/回撤比。

## 1. 实盘效果应如何定义与评估

实盘效果不能只看“胜率”，建议采用四层指标：

### 1.1 收益质量
- 年化收益（CAGR）
- 信息比率（IR）
- Alpha（相对基准）

### 1.2 风险质量
- 最大回撤（MDD）
- 波动率（Vol）
- 下行波动率、Calmar 比率

### 1.3 执行质量
- 滑点（bps）
- 成交偏离（VWAP/TWAP偏离）
- 指令完成率与拒单率

### 1.4 稳定性
- 月度胜率（rolling）
- 因子/规则贡献稳定性
- 分市场（A/H/US/QDII）分桶表现

## 2. v3.0 总体架构（优化点）

相较 v2.0，新增四个关键层：

1. **策略编排层（Strategy Orchestrator）**：统一路由与优先级冲突处理；
2. **回测与仿真层（Backtest/Replay）**：所有规则先离线验证再上线；
3. **风险闸门层（Risk Gate）**：硬性规则前置，交易信号后置；
4. **可观测性层（Observability）**：日志、指标、告警、审计链路。

架构原则：
- **研究与执行分离**：Claude 给研究意见，不直接下单；
- **规则先于观点**：铁律规则优先级最高；
- **配置即策略**：阈值、路由、仓位上限全部配置化；
- **先仿真再实盘**：没有通过验收指标不得切实盘。

## 3. 数据与特征工程规范

### 3.1 数据源分级
- P0（交易关键）：行情、IOPV、基金净值、汇率、公司公告；
- P1（增强）：资金流、估值分位、持仓披露；
- P2（解释）：新闻、研报摘要、社媒情绪。

### 3.2 数据质量规则（必须实现）
- 时间戳统一 UTC 与交易所本地时区双存；
- 关键字段缺失率 > 2% 触发降级；
- 同标的多源价差 > 1% 触发人工复核；
- IOPV/NAV 超过合理区间触发熔断（仅告警不交易）。

### 3.3 特征快照
每次决策写入 `feature_snapshot`：
- 原始输入值
- 清洗后值
- 最终用于规则判定值
用于审计与复盘。

## 4. 规则引擎升级（从 if-else 到可审计 DSL）

### 4.1 规则优先级
- L1 铁律（不可覆盖）
- L2 风控（可参数化）
- L3 机会（信号）
- L4 提醒（信息）

### 4.2 冲突处理
当“机会信号”和“风险信号”冲突：
- 以高优先级规则覆盖低优先级；
- 输出冲突原因（human-readable）；
- 记录 `blocked_by_rule_id`。

### 4.3 规则示例（伪DSL）
```
rule R_QDII_PREMIUM_KILL:
  when instrument.type == "QDII_ETF" and premium_pct > 13
  then action = "FORCE_SELL_ALL"
  level = L1
```

## 5. 回测/验证框架（上线前闸门）

### 5.1 三阶段验证
1. **历史回测（Backtest）**：至少覆盖牛/熊/震荡三类区间；
2. **事件回放（Replay）**：对极端日期做逐日回放；
3. **纸面交易（Paper Trading）**：连续 4~8 周稳定运行。

### 5.2 最低验收标准（建议）
- 年化收益 > 基准 + 5%；
- 最大回撤 < 12%；
- Calmar > 1.0；
- 月度正收益月份占比 > 60%；
- 关键告警漏报率 < 2%。

> 任一条未达标：不得进入实盘。

## 6. 模拟盘/实盘切换设计（提前规划）

### 6.1 账户抽象
统一接口 `BrokerAdapter`：
- `get_positions()`
- `place_order()`
- `cancel_order()`
- `get_fills()`

### 6.2 环境隔离
- `ENV=paper`：仅记录指令，不真实成交；
- `ENV=live`：真实交易，必须双重确认（人机）；
- 相同策略参数，不同执行器。

### 6.3 防误触机制
- live 模式必须满足：
  - 风控自检通过；
  - 当日数据完整；
  - 人工确认口令。

## 7. Claude 与飞书协同优化

### 7.1 Claude 职责边界
Claude 用于：
- 研究解释、情景分析、催化剂推演、反方观点构建。

Claude 不用于：
- 直接产生不可审计的交易指令；
- 覆盖铁律风控规则。

### 7.2 飞书卡片标准化字段
统一输出：
- 当前状态（🟢🟡🟠🔴）
- 触发规则 ID
- 三条关键数据（值+来源+时间）
- 单句操作建议
- 风险提示
